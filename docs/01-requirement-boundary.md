# 01. 需求边界怎么固化

真实采集项目开始前，先把需求边界写到纸面上，再动脚本。

## 为什么要先固化

评论采集类任务很容易在执行中变形：

- 对方说尽量多，但没有说多少算合格。
- 对方说有效评论，但没有说什么算无关。
- 对方给了样例 JSON，但没有说哪些字段必填。
- 对方要近期数据，但没有明确开始和结束时间。
- 对方要求按关键词搜索，但平台用户实际使用别名。

如果不先固化，后面会出现两个问题：

- 你以为自己完成了，对方觉得不可用。
- 数据客观低量时，你没有证据解释。

## 需求澄清清单

执行前至少确认这些问题。

| 问题 | 为什么重要 |
|---|---|
| 输入清单是什么 | 决定按什么目标归档 |
| 是否允许扩展关键词 | 决定搜索召回率 |
| 输出格式是什么 | 决定数据结构和交付脚本 |
| 每条记录必须有哪些字段 | 决定校验规则 |
| 时间范围是什么 | 决定过滤逻辑 |
| 单目标期望数量是多少 | 决定是否需要补采 |
| 冷门目标怎么处理 | 决定低量说明 |
| 什么算有效评论 | 决定文本过滤 |
| 是否需要保留来源 URL | 决定可回查性 |
| 是否先交样例 | 降低全量返工风险 |

## 推荐写成这样的边界

```md
## 已确认需求

- 平台：短视频网页端公开可见内容。
- 输入：目标名称列表，一行一个。
- 输出：JSON，按目标名称归档。
- 结构：目标 -> 视频 -> 评论。
- 必填字段：目标名、视频 ID、视频 URL、评论 ID、评论内容、评论时间。
- 时间范围：YYYY-MM-DD 到 YYYY-MM-DD，包含边界。
- 单视频限制：每个视频最多保留 N 条评论。
- 单目标期望：每个目标尽量 N 条有效评论。
- 低量口径：公开可见内容不足时按实际采集结果交付，并在报告中说明。
- 过滤口径：去重、去表情包、去明显无关内容、去明显广告。
- 交付节奏：先交 1 个目标样例，确认后全量。
```

## 低量目标要提前约定

不要承诺每个目标一定补满。

更稳的说法：

```text
热门目标会优先补足目标量。冷门目标、新车型、概念产品或平台公开内容较少的目标，以实际可采集到的相关公开评论为准。交付时会提供低量目标说明，不伪造、不补全不存在的数据。
```

这个口径看起来保守，但它保护的是交付可信度。
