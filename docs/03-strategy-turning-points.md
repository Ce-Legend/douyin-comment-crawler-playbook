# 03. 策略转向记录

一次数据项目的价值，既在最终文件，也在中间怎么转向。

## 转向 1：先样例，后全量

一开始就全量跑，会把格式错误、字段缺失、时间范围错误放大到所有数据里。

更好的节奏：

1. 先选 1 个目标。
2. 交付样例 JSON。
3. 让对方确认字段、结构、时间、URL。
4. 再跑全量。

这个动作看起来慢，实际减少返工。

## 转向 2：从自动搜索到真实页面

自动搜索适合低风险场景，但在登录态、验证码、动态加载明显影响结果时，要优先相信真实页面。

推荐策略：

- 人工打开目标页面。
- 确认页面上有相关视频。
- 脚本连接当前浏览器页面。
- 跑少量目标看有效数。
- 再扩大范围。

核心原则：先验证入口，再验证产量。

## 转向 3：从精确关键词到别名词表

客户给的输入名称是最终归档名，不一定是最优搜索词。

推荐建立一张映射表：

```csv
canonical_name,search_keyword,reason
Sample Model A,Sample Model A,original input
Sample Model A,Brand A Model A,platform title pattern
Sample Model A,ModelA,common short name
```

执行时：

- 搜索可以用多个关键词。
- 落盘仍用 `canonical_name`。
- 报告里说明哪些目标用了别名。

这样既尊重客户输入，又提升召回。

## 转向 4：从“继续跑”到“解释低量”

低量不一定是失败。

可能原因：

- 目标确实冷门。
- 公开可见视频少。
- 评论时间范围太窄。
- 平台搜索结果弱。
- 名称有歧义。
- 目标是概念产品或未上市产品。

低量目标应该进入报告，不应该用无关评论凑数。

## 转向 5：从结果文件到交付包

最终交付建议包含：

- `comments_full.json`：完整结果。
- `by_model/`：按目标拆分。
- `summary_by_model.csv`：每个目标统计。
- `low_volume_notes.csv`：低量目标说明。
- `field_dictionary.md`：字段说明。
- `quality_report.md`：校验报告。
- `README.md`：交付说明。

这能让对方不用读代码，也能判断结果是否可用。
