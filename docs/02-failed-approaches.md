# 02. 失败路线怎么判断

这个项目最有参考价值的地方，是第一版方案看起来能跑，但结果不可信。

## 常见误区

很多采集项目会把下面几件事误判成成功：

- 进程正常结束。
- 日志没有报错。
- 输出文件生成了。
- 速度很快。
- 数据结构看起来完整。

这些只能证明脚本跑完了，不能证明数据可信。

## 这次出现的危险信号

第一版自动化搜索路线出现了这些现象：

- 大量目标返回 0。
- 部分常识上应该有内容的目标也返回 0。
- 日志没有明显报错。
- 速度异常快。
- 页面状态和脚本结果不一致。

这时先按入口可能失真处理，暂停把 0 当成真实缺失。

## 什么时候该停下来

遇到下面任一情况，应该暂停全量：

| 信号 | 可能原因 |
|---|---|
| 热门目标也是 0 | 搜索入口失效、登录态异常、页面空壳 |
| 大量目标瞬间跑完 | 没有真正加载内容 |
| 新开页面结果少，已登录页面有内容 | 新会话风控或空页面 |
| 日志正常但数据不增长 | DOM/API 选择器失效 |
| 同目标前后差异极大 | 搜索结果不稳定或验证码干扰 |
| 页面肉眼可见有视频，脚本拿不到 | 抓取入口不对 |

## 最小验证实验

不要直接重跑全量。先做 3 个小实验：

1. 选一个常识上高热度的目标，人工打开页面确认是否有内容。
2. 用同一个已登录页面跑脚本，看有效数据是否增长。
3. 对比自动新开页面和复用人工确认页面的结果差异。

如果人工确认页面有内容，而自动路线拿不到，优先修入口，不要继续堆并发。

## 复盘句式

可以把失败路线写成这样：

```md
第一版自动化搜索方案可以生成输出，但大量目标返回 0，且部分热门目标也没有数据。这个结果和平台内容常识冲突，因此我回到页面层验证，没有直接把 0 当成真实缺失。验证后发现，复用已登录真实搜索页能获得更稳定的数据增长，所以后续放弃盲跑批量搜索，改成小批量验证 + 低量补采。
```
